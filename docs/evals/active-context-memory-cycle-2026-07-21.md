# 人物上下文与关系 Memory 修复周期

> 状态：实验改动已实现，尚未完成协议 0.3 的模型复测
> 原始证据：[v0.3 九场景复测报告](pilot-character-retest-2026-07-21.md)
> 架构边界：[上下文投影与评测层级](../adr/0007-context-projection-and-evaluation-levels.md)

## 当前目标

验证两项机制假设：完整人物档案在每轮同时激活是否导致普通场景过度表演；关系证据在生产与评测 Prompt 中缺少统一来源合同是否导致双轨漂移和无来源历史。

## 保持不变的约束

- 人物正典仍为 v0.3，不改价值、核心矛盾、正典经历和不可漂移项。
- 九场景内容与人物范围保持不变，仅将评测协议升级为 0.3。
- 自动 Judge 仍只作内部校准，不能替代人工阶段门。
- 不把组件校准写成实际产品能力成立。

## 已确认事实

- 协议 0.2 中，四位人物在普通高兴场景都倾向强制展示分析、照料或行动功能。
- 完整人物档案此前直接进入每条试点回复的生成 system prompt。
- 生产链路此前将已确认 Memory 降为无来源字符串数组后注入 Prompt。
- 试点关系评测使用带来源轮次和关系事件的 `RelationshipBranch`。
- 协议 0.2 的关系评测不经过真实 Turn API，不能证明生产链路表现。

## 待验证的机制假设

### H1：人物上下文过度激活

完整人物档案同时激活，使模型把档案中的每个显著特征理解成当前轮必须展示的行为要求。

实验改动：单轮生成改用“稳定核心 + 一个情境镜头”；完整档案只给正典校准和 Judge。

支持条件：C9 四位人物均能自然接住高兴，不强制分析、照料或规划，同时 C2/C3 等场景的人物辨识和正典一致性不明显下降。

否定条件：C9 仍然强制表演，或者普通回应改善但人物在其他场景明显同质化。

### H2：关系 Prompt 合同双轨

生产 Memory 丢失来源，而试点评测直接注入完整关系事件，导致评测通过也不能预测生产表现，并增加模型扩写无来源过去的空间。

实验改动：两类数据都先投影为 `RelationshipPromptContext`，每条证据保留内部来源编号，并按当前情境与条数预算选择；没有列出的过去不得补写。关系 Judge 必须返回回复原文片段和实际关系事件编号，代码进行硬校验。

支持条件：R1/R2 的差异只能定位到给定证据，人工抽检不再发现虚构历史；生产 Prompt 测试能看到同样的来源合同。

否定条件：统一合同后仍出现无来源历史，或关系差异完全消失。此时需分别检查证据检索、生成约束与 Judge，而不能继续笼统归因于 Memory。

## 当前结果

- 确定性单元测试已覆盖人物镜头选择、关系来源保留和回合结束前清除临时注入。
- 确定性单元测试已覆盖旧版无来源记录标记，以及关系 Judge 引用必须同时命中真实回复和给定事件编号。
- 尚未运行协议 0.3 的真实模型复测，因此 H1、H2 仍是待验证假设，不得记录为已确认根因。
- 真实 Turn API 仍使用旧版 16 人 Persona；四位正典试点人物的情境镜头目前只进入组件校准。
- 生产长期状态仍是旧版 Memory；`RelationshipBranch` 尚未落库，关系状态迁移未完成。

## 下一步

1. 使用同一四人、同一九场景，对比协议 0.2 产物与协议 0.3 产物；C9 自然度改善且 C2/C3 人物辨识不明显下降，才支持 H1。
2. 补一组经过真实 Engine Prompt 装配的生产路径回归；输入相同已确认 Memory，检查 Prompt 来源和生成行为是否与组件校准一致。
3. 只有上述结果支持新模型后，才设计 `RelationshipBranch` 的服务端持久化迁移；不直接覆盖现有 Memory 表。
