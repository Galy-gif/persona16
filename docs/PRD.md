这份 PRD 是 persona16 的产品总装文档，不是页面清单。它按以下依赖顺序组织：

> 业务场景 → AI 必要性 → L1–L8 主链路 → 评测体系 → 功能与状态 → 交互与视觉 → MVP 阶段门

文档中的结论使用四种状态：

| 状态 | 含义 |
| --- | --- |
| **已确认** | 用户已经明确确认，可作为产品与工程依据 |
| **方向确认** | 方向已被接受，但细节仍需讨论，不能直接完成最终实现 |
| **待验证** | 当前是假设，需要样本、评测或真实用户证据 |
| **待讨论** | 尚未做产品决策，不得由实现者自行补全 |

本轮只重构 PRD 逻辑，不把待讨论内容包装成既定需求。

## 1. 项目概述与功能锚定

### 1.1 一句话定义｜已确认

persona16 是一个移动端优先的人格对话产品。用户从 16 位性格稳定、会逐渐了解自己的 Agent 朋友中选择一位开始聊天，也可以在对话中邀请其他 Agent 加入房间，从不同人格视角看见自己原本没有想到的角度。

### 1.2 用户任务卡

| 项目 | 当前结论 | 状态 |
| --- | --- | --- |
| 目标用户 | 想通过聊天获得不同视角、思想碰撞或情感陪伴的人；不要求理解完整 MBTI 理论 | **已确认** |
| 首批用户画像 | 年龄、使用 AI 习惯、人格文化熟悉度和招募渠道 | **待验证** |
| 触发时刻 | 用户有日常闲聊、无厘头问题、职业/关系/情绪困惑，或单纯想找一个熟悉的 Agent 说话时 | **已确认** |
| 用户输入 | 自然语言消息，以及点名、邀请、暂停、反馈、记忆确认等明确控制动作 | **已确认** |
| 当前卡点 | 通用 AI 容易同质化、写论文、强行分析或假装深刻；用户难以获得稳定而不同的人格视角 | **已确认** |
| 用户期望结果 | 得到一段愿意继续接话的自然对话，并发现一个不同视角；安慰、建议或行动方案不是每次强制输出 | **已确认** |
| 失败成本 | 用户觉得“AI 味重”、人格在表演、被无依据读心或被情绪操控，从而不愿继续聊 | **已确认** |

### 1.3 核心价值｜已确认

> 让用户借用一种不同的人格视角，看见自己原本没有想到的角度。

产品不替用户寻找统一的标准答案。16 个 Agent 是 16 位朋友，不是 16 个功能模块。用户聊得越久，关系可以更熟悉，但不能通过内疚、孤立或依赖话术操控用户。

### 1.4 第一版要证明什么｜待验证

MVP 需要形成以下最小证据闭环：

1. 用户能选择一位 Agent 并完成一轮自然对话。
2. 不同 Agent 对同一输入表现出可辨认但不刻板的视角差异。
3. 用户愿意继续追问、换一位 Agent 重问，或之后再次找同一位 Agent。
4. 多 Agent 房间确实产生补充、分歧和化学反应，而不是多人轮流作文。
5. 差评和 bad case 可以回到某个链路节点，并用同一批样本复测。

“用户获得新角度”和“愿意回来找同一个 Agent”的具体量化口径仍待校准。

## 2. 为什么需要 AI

### 2.1 非 AI 方案可以解决什么｜已确认

规则、数据库和普通产品逻辑负责：

- Agent 分组、选择、邀请、暂停、移除和人数上限；
- 房间权限、消息历史、幂等、限流、预算和硬停止条件；
- 记忆的确认、查看、关闭和删除；
- 输出协议、反馈字段、评测样本和版本记录；
- 高风险内容的确定性兜底与独立安全流程。

### 2.2 非 AI 方案解决不了什么｜已确认

用户输入是开放的自然语言，同一句话会因语境、关系和说话对象不同而产生不同含义。固定模板难以同时完成：

- 理解用户此刻更需要闲聊、追问、反驳、安慰还是沉默；
- 在稳定人格下产生不重复、不过度表演的自然表达；
- 根据关系阶段改变直接程度和互动节奏；
- 在多人房中判断谁此刻有新增价值、谁应该保持沉默；
- 从对话中提出可能值得长期记住的信息。

### 2.3 AI 介入节点｜已确认

AI 只在难以标准化的判断与生成节点介入：

1. **L3 上下文理解**：理解开放输入、场景、情绪与缺失信息。
2. **L4 人格化生成**：结合人格核心、运行时状态、关系和上下文生成回应。
3. **L4 房间调度建议**：提出下一位说话者、发言类型和收束建议，再由确定性规则校验。
4. **L7 候选记忆提取**：提出候选事实或偏好，但必须由用户确认后才能保存。
5. **评测辅助**：按已有人类 rubric 批量初评，不能未经校准替代人工判断。

### 2.4 AI 不负责什么｜已确认

- 不做心理诊断，不替代医疗、法律或危机支持。
- 不决定未经用户确认的长期事实。
- 不自行突破房间人数、预算、安全和停止条件。
- 不以 MBTI 标签或中文代号直接推导固定台词。
- 不为了保持关系制造内疚、占有感或依赖压力。

## 3. MVP 唯一主链路（L1–L8）

### 3.1 主链路一句话

> 这条链路把“用户选择一位想聊的 Agent 并说出当下的话”，经过“安全与上下文准备、人格化单聊或有限多人房处理”，变成“自然、有差异且可继续的对话”，再通过反馈、确认式记忆和关系状态影响下一次聊天。

### 3.2 主链路总表

| ID | 环节 | 核心问题 | 第一版处理 | 输出 | 主要失败与兜底 |
| --- | --- | --- | --- | --- | --- |
| L1 | 触发 | 用户为什么此刻开始，以及先找谁 | 从首页选择一位 Agent 新建话题；或从历史恢复旧话题 | 新话题/旧话题 + 初始成员 | 16 人过载：先分四组，再选一人；不自动创建多人房 |
| L2 | 输入 | 系统实际收到什么 | 用户文本、房间 ID/版本、点名或成员控制动作 | 可信的用户命令 | 空文本、超长、重复、过期版本或越权：规则拒绝并提示恢复 |
| L3 | 预处理 | 本轮可否执行，需要哪些上下文 | 权限、幂等、限流、安全分流、上下文裁剪、已确认记忆选择 | 最小可用 TurnContext | 高风险走独立安全流程；上下文不足时追问；未确认记忆不得注入 |
| L4 | 核心处理 | 谁说、怎么说、是否继续 | 单聊调用人格运行时；多人房由控制器提议、确定性策略校验、Pi 执行选中 Agent | 有序发言、追问、总结或停止 | 重复、全员长答、循环、超预算：有限循环与硬停止；执行失败可恢复 |
| L5 | 输出 | 用户如何看到并继续对话 | 流式输出 Agent 身份、文本、状态和可恢复错误 | 一轮可消费的对话结果 | 空回复、断线、超时：取消、重试或明确的 `DELIVERY_FAILED`，不伪造完整结果 |
| L6 | 反馈 | 如何知道这一轮哪里好或坏 | 单条反馈、原因标签、取消/重生成、点名、换 Agent、记忆确认 | 结构化反馈事件 | 继续聊天不直接等于满意；所有指标保留版本和样本条件 |
| L7 | 状态沉淀 | 什么值得影响下一次 | 会话事实、关系状态、用户确认记忆、运行 trace、bad case 和版本 | 可追溯、可删除的状态 | 无用途数据不保存；反馈与评测数据不直接当人格记忆 |
| L8 | 下一次 | 历史状态如何推动下一次服务 | 新建话题、恢复历史、再次找同一 Agent、邀请其他 Agent | 新一轮 L1 | MVP 不主动推送；未来主动联系必须授权、可关闭且不施压 |

### 3.3 逐节点下钻模板

接下来每次只完善一个 L 节点。每个节点必须回答：

1. 输入可能有哪些情况？
2. 处理动作如何拆分？
3. 哪些使用规则、数据库、人工、Prompt 或 Agent？
4. 第一版选择什么实现方式，为什么？
5. 输出是什么，交给谁？
6. 保存什么状态，下次怎么用？
7. 最容易在哪里失败，如何提醒、拒绝、追问、重试或转安全流程？
8. 用什么样本和指标判断这个节点是否合格？

未完成以上八项的节点，不能仅以“调用大模型”作为技术说明。

## 4. 从主链路推导出的功能

功能必须能挂回一个或多个链路节点。挂不回主链路的功能不进入 MVP。

| 功能 ID | 功能模块 | 对应节点 | 第一版职责 | 状态 |
| --- | --- | --- | --- | --- |
| F1 | 新话题入口与 Agent 选择 | L1 | 先选一位 Agent，再创建干净的新话题 | **已确认** |
| F2 | Agent 身份与四组导航 | L1 | 中文代号为主、MBTI 弱标签；16 人先分四组，再进入第二层选择 | **方向确认** |
| F3 | 单 Agent 对话 | L2–L5 | 自然语言输入、流式回复、取消、错误恢复和重新生成 | **已确认** |
| F4 | 多 Agent 房间 | L1、L4–L5 | 单聊中邀请其他 Agent，最多 3 位；允许回应、补充、质疑和沉默 | **已确认** |
| F5 | 历史与多话题 | L1、L7–L8 | 新话题独立保存；历史按时间恢复；首页选择默认新建话题 | **已确认** |
| F6 | 回复反馈与 bad case | L6–L7 | 有帮助/不太对、原因标签、版本与 trace 归因 | **已确认** |
| F7 | 关系与确认式记忆 | L3、L6–L8 | 自动关系状态 + 用户确认的具体长期事实；候选确认任务按房间恢复；支持关闭和删除 | **已确认** |
| F8 | 游客身份与数据 | L1–L3、L7 | 游客优先，当前设备保存；登录与跨设备后续设计 | **已确认/细节待讨论** |
| F9 | 安全、预算与运行观测 | L2–L7 | 独立安全旁路、权限、限流、硬预算、事件和 fail-open trace/observer | **已确认** |

### 4.1 不从页面反推功能

首页、抽屉、底部导航和渐变背景是承载 F1/F2/F5 的交互与视觉方案，不是独立产品价值。后续页面设计必须说明自己服务哪个功能和链路节点。

## 5. Agent 身份与人格产品规则

### 5.1 正式代号｜已确认

| MBTI | 代号 | MBTI | 代号 |
| --- | --- | --- | --- |
| INTJ | 笑面虎 | INTP | 慢半拍 |
| ENTJ | 大当家 | ENTP | 反骨 |
| INFJ | 随身听 | INFP | 犟种 |
| ENFJ | 操心命 | ENFP | 人来疯 |
| ISTJ | 老干部 | ISFJ | 顺毛 |
| ESTJ | 不倒翁 | ESFJ | 东道主 |
| ISTP | 闷葫芦 | ISFP | 散人 |
| ESTP | 乐子人 | ESFP | 显眼包 |

中文代号用于身份识别和第一印象，不是人格行为指令。不能为了贴合名字，把 Agent 固定成单一性格梗、口头禅或表演模板。

### 5.2 人格生成原则｜已确认

- 人格是稳定核心 + 运行时动态状态 + 当前关系与语境。
- 人格核心定义“更容易注意什么、如何介入”，不堆叠形容词或文学化声线。
- Agent 自己决定何时直接回答、追问、反驳、安慰、道歉或沉默。
- 宁可让人格表现稍弱，也不能让用户感觉 Agent 在努力表演人格。
- 多人房通过发言选择、沉默、短补充和分歧产生价值，不是所有角色轮流回答。

### 5.3 自然口语合约｜已确认

- 默认使用日常中文，一次发言聚焦一个主要角度。
- 回复长度由问题、人格和关系决定，但必须给用户留下接话空间。
- 不默认写论文，不大量使用抽象词、隐喻和强行升华。
- 不无依据判断用户的深层动机。
- 不依赖“你不是 X，而是 Y”等句式制造假洞察。
- 抽象判断应回到具体事实、问题或行动。

## 6. 模型、运行时与 Prompt / Schema 管理

### 6.1 方案边界｜已确认

- 底层通用运行时采用开源 Pi Agent，以 adapter 接入 `pi-agent-core` / `pi-ai`。
- Pi 负责单 Agent 模型调用、事件流、取消、重试和运行时状态。
- persona16 负责人格 DNA、动态状态、房间编排、关系记忆、安全策略和评测。
- MVP 不开放浏览器、Shell、文件写入等通用工具。
- 多 Agent 房间循环不能交给 Pi 自由运行，必须由有限房间策略控制。

### 6.2 产品级 Prompt / Schema 注册表｜待补全

完整 Prompt 正文不贴在 PRD 中，但每个 AI 节点必须登记：

| Prompt/规则 ID | 使用节点 | 输入变量 | 输出 Schema | 核心约束 | 失败兜底 | 当前状态 |
| --- | --- | --- | --- | --- | --- | --- |
| SAFETY | L3 | 用户输入、最小上下文 | 安全等级与处理路径 | 独立于人格 | 保守分流 | **已有实现，待文档化** |
| PERSONA | L4 | 人格核心、关系、房间状态、用户输入 | 自然语言发言 + 运行元数据 | 自然口语、非模板、无读心 | 重试/可恢复错误 | **已有实现，待文档化** |
| ROOM_CONTROLLER | L4 | 成员、最近发言、预算、用户指令 | action-specific 的 speak/ask/summarize/stop 联合类型 | 每种动作只含必要字段，提议仍由规则校验 | Schema 解析失败则结束当前执行并返回可恢复错误 | **窄 Schema 已实现** |
| ROOM_COMMAND_TOOL | L2、L4 | 模型表达的窄房间控制意图 | `pause_agent({ agent })` 等单动作合同 | 模型不能选择 roomId、userId、版本或绕过 RoomPolicy | 未接入前保持 `tools=[]` | **合同基础已实现，执行链路后置** |
| MEMORY_CANDIDATE | L7 | 已完成回合 | 候选事实/偏好 | 不自动写入 | 用户确认或丢弃 | **已有实现，待文档化** |
| EVAL_JUDGE | 评测 | eval case、rubric、版本 | 分数、理由、失败类型 | 先人工校准 | 人工抽检 | **部分完成** |

所有记录必须包含 `prompt_version`、`rubric_version`、`model_version` 和可复测的运行条件。

### 6.3 窄工具合同与 Room Command 边界｜工程基础已完成，产品接入后置

当前已完成的基础：

- RoomController 使用 action-specific 的 Zod discriminated union；`speak`、`summarize`、`ask_user`、`stop` 只携带各自动作需要的字段，`stop.reason` 使用固定枚举，不再从自然语言猜测停止原因。
- `defineRuntimeTool` 以 Zod 为输入合同的单一事实源：同一份 Schema 生成模型可见 JSON Schema，并在执行 handler 前完成运行时解析；handler 自动获得强类型参数。
- 已定义 `pause_agent({ agent })` 窄合同与结构化结果类型。模型只能表达要暂停的 Agent；`roomId`、`userId` 和房间版本必须由未来 Harness 从可信上下文注入。
- 已实现统一 RoomCommand 模块，现有 UI 房间操作通过该模块执行所有权、Permission、RoomPolicy、版本和忙碌状态检查；接口拆为 `pause_agent`、`resume_agent`、`invite_agent`、`remove_agent` 四个窄动作。

这些基础不代表产品已经开放 Agent 工具调用。当前 Persona 请求继续保持 `tools=[]`、`maxTurns=1`，`pause_agent` 没有接入 `runTurn`；只有用户直接操作 UI 时会通过 RoomCommand 模块修改房间。

后续进入产品链路前必须另行完成：

1. 明确 Room Command Harness 位于 L2/L3 预处理还是独立命令入口，不能把持久房间管理权直接交给 16 个 Persona。
2. 在未来 Harness 中复用现有 RoomCommand 模块，从服务端注入用户、房间、版本和授权依据；不得在工具执行器中重写房间规则。
3. 将 `resume_agent`、`invite_agent`、`remove_agent` 继续拆成单动作合同；不提供 `manage_room(action, data)` 或 `toggle_agent`。
4. 设计自然语言触发、用户确认、命令审计、幂等与失败恢复，再用固定正负样本评测该调用/不该调用。
5. 只有上述评测和安全门通过后，才为专用 Room Command 流程传入工具并提高 `maxTurns`；Persona 默认生成路径不自动继承这些工具。

### 6.4 Room Command Permission｜服务端基础已实现，聊天确认流后置

Permission、RoomPolicy 和并发控制必须分层：Permission 判断发起方是否获得用户授权；RoomPolicy 判断当前房间是否允许该变化；版本与忙碌检查判断授权所依据的状态是否仍然有效。Prompt 中的“请谨慎调用”不能替代服务端 Permission。

| 授权来源 | 暂停/恢复 | 邀请 | 移除 |
| --- | --- | --- | --- |
| 用户明确点击 UI | 允许 | 允许 | 二次确认后允许 |
| 用户在聊天中明确表达 | 允许 | 允许 | 二次确认后允许 |
| 模型根据抱怨、语气或上下文自行推断 | 拒绝 | 拒绝 | 拒绝 |
| Persona 自己建议或 Safety 系统触发 | 仅可提出建议，不修改房间 | 拒绝 | 拒绝 |

当前实现链路：服务端 Session 确认用户 → Store 隐藏式校验房间所有权 → RoomCommand 检查授权来源 → 检查预期版本/忙碌状态 → 检查成员与房间规则 → Store 原子比较并更新。客户端和模型都不能自行提交 `userId` 或把 `permission=allow` 写进工具参数。

聊天中的自然语言授权尚未接入。后续需要先生成可审计的 `explicit_user_text` 授权依据；模糊表达必须拒绝调用或向用户澄清，不能降级为 `model_inference` 后继续执行。

## 7. 状态、关系与记忆

### 7.1 只保存能被使用的状态｜已确认

状态至少进入以下一种用途，否则不应保存：

- 作为下一次生成的必要上下文；
- 作为用户表达偏好的依据；
- 作为 bad case 或同批复测样本；
- 作为 Prompt / rubric 迭代依据；
- 作为未来经用户授权的主动触发条件。

### 7.2 两层记忆｜已确认

1. **关系状态**：熟悉程度、表达直接程度和互动节奏可以随交互自动变化。
2. **长期事实与偏好**：职业、关系、经历和表达偏好等具体信息，必须由用户确认后保存。

候选记忆同时是一项等待用户处理的工作，而不只是数据库记录。已完成回合产生的 `candidate` 必须按来源房间恢复；刷新、重新进入或手动刷新房间后，用户仍能继续“确认记住 / 不记”。其他房间的候选不得出现在当前房间。只有 `confirmed` 能进入 Prompt。

关系变化不显示为数值、等级或进度条。用户通过 Agent 的表达变化自然感受到关系发展。

### 7.3 隐私与共享边界｜已确认

- 某条信息默认只属于听到它的 Agent。
- 其他 Agent 不会因为同属 persona16 自动知道这条信息。
- 当 Agent 被邀请进房间时，可以带入与当前话题相关的上下文。
- 房间参与者记得自己参加过的对话；具体长期事实仍需用户确认。
- 用户可以查看、关闭和删除长期记忆。
- 不把密钥、证件号码、银行卡等敏感信息保存为人格记忆。

记忆开关采用全局、单 Agent 还是单话题粒度仍为 **待讨论**。

### 7.4 循环外围能力的失败合同｜已确认

- Core/Policy（所有权、Permission、版本、安全、房间预算、最终持久化）失败时终止或拒绝执行，默认 fail closed。
- Delivery（版本化 lifecycle 事件与流式 `delta`）失败必须转换为明确错误或显式取消，不得被静默吞掉；消息 ID 由核心执行结果生成，不依赖投递回调补写。
- Observer（trace、metrics、analytics 和非关键生命周期副本）失败时隔离并记录，不得中断 Agent Loop；连续 trace 写入失败后关闭当前 tracer。
- 在出现第二个真实外部观察消费者前，不建设通用 Hook Bus，继续使用窄类型回调。

## 8. 评测体系与 Bad Case 闭环

### 8.1 评测目的｜已确认

第一版评测主要面向内部迭代，回答三个问题：

1. AI 输出到底好不好？
2. 如果不好，问题发生在 L1–L8 的哪个节点？
3. 修改后，能否用同一批样本和同一 rubric 证明它变好？

当前没有足够真实用户证据，因此不声称留存、满意度、增长或长期关系价值已经提升。

### 8.2 两层评测对象

| 层级 | 评测什么 | 目的 | 当前例子 |
| --- | --- | --- | --- |
| A. 链路节点诊断 | 输入、上下文、人格生成、房间调度、输出、反馈、状态与安全 | 定位应该修改哪个节点 | runtime regression、安全测试、房间对抗、trace |
| B. 端到端用户价值 | 是否出现不同视角、是否自然、是否愿意继续接话、多人房是否有新增价值 | 判断用户任务是否真的被解决 | 人格盲测、动态性、房间化学反应、自然度人工评测 |

### 8.3 Eval case 结构｜方向确认

第一版评测集应稳定、可复测，并覆盖：

- 高频常规 case；
- 口语化、模糊或无厘头 case；
- 易混淆人格与易同质化 case；
- 边界、高风险、冲突和隐私 case。

每条 case 至少记录：`case_id`、`case_type`、`stage_id`、输入、期望要点、系统输出、rubric 得分、低分维度、失败类型、trace 归因、模型/Prompt/rubric 版本、修改动作和复测得分。

### 8.4 当前指标与待补指标

| 对象 | 当前指标 | 当前边界 |
| --- | --- | --- |
| 人格辨识 | blindtest top-3 辨识、同质化 | 已有离线自动/人工页面，仍需独立人工校准 |
| 动态性 | 同一 Agent 在多种关系与房间场景下改变策略 | 已有离线评测 |
| 房间价值 | 分歧、补充、收束、沉默与硬停止 | 已有基础与对抗评测 |
| 自然口语 | 真人口语感、AI 味、读心、假深刻、长度、是否愿意接话 | rubric 与样本仍待补全 |
| 核心用户价值 | 是否发现新角度、是否愿意再次找该 Agent | **待定义和真实用户验证** |
| 在线行为 | 换 Agent、同题复玩、继续追问、恢复历史、回访 | 仅作为后续观察，不声称已有提升 |

自然度初始守门目标：总反感率不高于 12%，且人格辨识度不低于当前基线。该阈值需要通过人工样本校准后才能成为正式发布门。

### 8.5 Bad case 复测规则｜方向确认

每个低分样本必须形成：

> 低分维度 → 对应链路节点 → trace 原因 → 修改动作 → 同批样本复测 → 新旧版本对比

不能用“模型不行”或“继续优化 Prompt”作为最终归因。

## 9. 交互与视觉承载

交互设计必须服务 L1–L8，不先于链路决定产品功能。

### 9.1 已确认的交互原则

- 用户必须先选择一位 Agent，才能开始新话题。
- 首次选择只允许一个 Agent；多人房从聊天中邀请形成。
- 选择后创建干净的新话题；从历史进入才恢复旧话题。
- 输入框只在聊天房间出现，不在选择页展示问题示例或示例回复。
- Agent 列表以中文代号为主，MBTI 是低视觉权重辅助标签。
- 首页不重复展示“最近聊过的人”；历史记录有独立入口。

### 9.2 首页与两层选择｜方向确认

当前讨论方向是：

1. 首页显示一句轻量运营文案和四个人格分组入口。
2. 四组以紫、绿、蓝、黄的低饱和融合色区分。
3. 点击分组后进入第二层选择面板；底部抽屉是当前优先方案。
4. 第二层展示该组 4 位 Agent；点击 Agent 后直接创建新话题并进入聊天。

仍需确认：四组的原创中文名称、四张入口的排列、抽屉高度和布局、历史/个人入口位置、返回与转场细节。

### 9.3 视觉方向｜方向确认

- 首页使用粉色与蓝色互相融合的低对比柔光渐变，不做明显线性渐变。
- 整体气质简约、舒适，不特别少女、小清新或老成。
- 四组入口采用轻量、低饱和、带空气感的彩色填充，接近粉彩但不糖果化。
- 聊天房间以白色或浅中性背景为主，减少长时间阅读负担。
- 渐变、磨砂和阴影是材质手段，不是独立装饰；避免强霓虹、厚重玻璃和高饱和大色块。

头像、人物形象、字体、精确色值、圆角、图标和最终布局仍为 **待讨论**。

## 10. MVP 范围与产品边界

### 10.1 MVP 必须支持｜已确认

- 16 个固定人格 Agent 及正式中文代号；
- 四组导航、选择一位 Agent、新建单聊话题；
- 单聊中邀请其他 Agent，转换为最多 3 人格房间；
- 邀请、移除、暂停、恢复和点名；
- Agent 之间回应、补充、质疑或沉默；
- 流式生成、取消、重新生成和错误恢复；
- 单条回复反馈与原因标签；
- 多话题历史记录和恢复；
- 关系状态、用户确认式记忆及关闭/删除；
- 游客优先；
- 独立安全层、有限循环、运行观测和评测回归。

### 10.2 当前不纳入｜已确认

- 官方 MBTI 测评；
- 用户自建人格和人格市场；
- 16 个 Agent 同时群聊；
- 固定“最佳人格组合”推荐；
- 强制登录；
- 语音聊天、数字人和复杂动效；
- 聊天记录卡片导出；
- 未经用户授权的主动通知；
- 浏览器、Shell、文件写入等通用 Agent 工具。

### 10.3 账号与主动联系

- MVP 采用游客优先，当前设备保存会话、关系与确认记忆。**已确认**
- 正式登录、游客数据迁移、跨设备同步与合并策略。**待讨论**
- 长期方向允许 Agent 主动联系用户，但必须明确授权、可关闭且不制造内疚。**方向确认**
- 主动联系是否进入 MVP、触发条件、频率和渠道。**待讨论**

## 13. 证据边界与文档优先级

### 13.1 当前可以声称

- 已实现人格引擎、Pi Runtime 适配、有限多人房、服务端状态、确认式记忆、安全旁路、运行 trace 和一组离线评测。
- 已确定核心产品价值、关系与记忆原则、16 个中文代号、单聊转多人房逻辑和自然口语方向。
- 已形成首页与视觉偏好，但尚未完成正式 UI 方案。

### 13.2 当前不能声称

- 真实用户留存、满意度、增长或商业化已经被验证；
- 长期关系价值、主动陪伴或回访效果已经成立；
- 现有自动 Judge 可以替代人工评测；
- 页面原型已经代表最终产品交互和视觉。

### 13.3 文档优先级

出现冲突时，按以下顺序判断：

1. 本 PRD 中标记为“已确认”的产品逻辑；
2. 本 PRD 的 L1–L8 主链路和评测口径；
3. [MVP 开发路径与技术方案](MVP-development-plan.md)中的工程边界与阶段门；
4. ADR 与实际测试证据；
5. 2026-07-05 原始 PRD 与 persona spec，作为研究与历史背景。

开放问题和方向偏好不构成最终需求，不得由实现者自行补全。
