# 试点房间采用人物私有参与意向与逐发言仲裁

## Context

协议 0.4 的隔离房间预检固定按 `ENFP → ESTP → INTJ → ISFJ` 生成，并强制 INTJ、ISFJ 发言。它能够检查强提示下的文本串联，却不能检查正典人物是否会主动不发言、谁适合先说，或一条公开发言是否会改变后续人物的参与判断。`responsibilityTransferCount` 又由 Judge 自由计数，无法可靠区分房间话轮控制与用户现实项目中的维护、回滚和停止责任。

ADR-0002 规定的是当前生产有限 Room Loop；ADR-0009 规定协议 0.4 的试点 Prompt、硬门和评测产物。本决策只替代 ADR-0009 的房间预检部分，不改变 ADR-0002，也不授权四位正典人物进入生产。

## Decision

### 人物提交房间参与意向

每个仍未发言的正典人物根据当前用户消息、已发生发言、正典人物核心与房间情境镜头，私下提交：

- `speak | brief_addition | pass`；
- 准备新增的 `claimSummary`；
- 可选的已有 `targetMessageId`；
- 不发言时的 `passReason`。

意向不是公开台词，也不包含可直接横向比较的自评分。`pass` 不产生 `【沉默】` 等用户可见消息。

### 房间仲裁器逐次授权

房间仲裁器只在当前合格意向之间比较直接相关性、边际新增价值和消息依赖，一次授权一个公开发言。每条消息写入 transcript 后，剩余正典人物重新提交意向；没有合格意向时以 `no_eligible_intent` 自然停止。

四人组件校准允许 1–4 人发言，不设固定必说人物或沉默配额。`maxVisibleActs`、assessment round、总时长与总字符只作为预算保险丝；触发后必须报告 `budget_exhausted`，不能记作自然沉默。

### 顺序与现实责任进入结构门

- `targetMessageId` 只能引用已存在消息，公开消息必须保留被选意向的同一依赖。
- 现实责任声明记录 activity、责任主体注册表 ID、主体类型、状态、当前陈述片段、逐字证据与来源消息。
- 正典人物和房间仲裁器在注册表中明确标为不可承担现实 activity；自由文本别名和未注册主体也不能成为 owner。
- 每条责任声明都必须有可回查 source 和 quote；同一回复中真正涉及归属的每个责任语句按 activity 分别建账，普通流程条件不被误判为责任转移。只有逐字证据在可见来源中唯一匹配时才允许确定性修正来源 ID；Judge 不再自由计算责任转移次数。

## Evaluation level

协议 0.5 的当前结果是**组件校准**。它直接运行隔离评测入口和模型调用，不经过真实 Turn API、生产 Store 或 ADR-0002 的 Room Loop，因此不是生产路径回归或端到端验证。

## Consequences

- 人物是否参与与房间如何维持秩序不再由同一个中央判断混写。
- 发言顺序通过消息依赖形成偏序，不需要固定唯一名单。
- 沉默比例只在多样本评测中作为诊断指标，不进入单轮配额。
- 责任缺口可以保持 `unassigned`；房间仲裁器不替现实团队确认负责人。
- 评测协议升级为 0.5，房间参与版本为 `pilot-room-participation-v0.1`；Prompt 装配仍为 `pilot-runtime-prompt-v0.4`。
- 生产 Room Loop 若要迁移此机制，必须另行设计生产路径回归、成本与延迟预算，并重新审视 ADR-0002，而不能从本组件校准直接推导上线能力。
