# 正典档案通过上下文投影进入生成，并区分评测层级

## Context

人物正典 v0.3 的四人 × 九场景复测暴露出两类相互关联的问题：完整人物档案在每轮生成中同时激活，普通场景容易变成人物功能表演；试点关系评测使用事件驱动的 `RelationshipBranch`，生产链路则从已确认 Memory 注入偏好、重复模式和边界。两条链路虽都在测试关系差异，但此前没有共享同一个 Prompt 输入合同。

因此，试点关系评测只能作为候选关系模型的组件校准，不能单独证明当前产品链路已经实现关系养成。

## Decision

### 人物上下文投影

- 完整正典人物档案继续作为版本化真相源，以及人工和 Judge 校准依据。
- 单轮生成不再默认加载完整档案，而使用“正典人物核心 + 一个当前情境镜头”。
- 正典经历、自我故事全量、全部关系方式和表达示例默认不进入单轮生成。
- 当前镜头可以是普通互动、决策、承托、分歧、修复、明确结束或多人房。
- 普通互动明确允许人物倾向不显影；明确结束镜头只允许简短确认并停止。

### 统一关系上下文合同

生产 Memory 与 `RelationshipBranch` 暂时可以保留不同的存储和状态更新方式，但进入 Prompt 前必须投影为同一个 `RelationshipPromptContext`：

- 每条证据包含内容、类别和内部来源编号；
- 当前情境决定证据类别优先级，并受固定条数预算限制；不能把全量关系档案重新塞回 Prompt；
- 来源轮次表示“用户在哪一轮说过或发生过”；
- 来源事件表示“该轮被记录成了哪一种关系变化”；
- 来源编号只用于内部校验，不向用户朗读；
- 没有进入上下文包的过去不得由模型自行补写；
- 旧版无来源记录必须显式标记为不可追溯，不能伪装成共同经历。

`RelationshipPromptContext` 是生成接口，不是新的长期真相源。生产 Memory 表和未来的关系分支存储仍由服务端负责。

### 评测层级

所有结论必须标明评测层级：

1. **组件校准**：直接构造人物或关系上下文，验证候选模型是否有潜力。
2. **生产路径回归**：调用 Engine 的真实 Prompt 装配和生成入口，但可使用固定 Store 或模型替身。
3. **端到端验证**：从真实 Turn API、服务端 Store、RoomLoop、生成、校验和持久化完整经过一次。

组件校准通过不能自动升级为产品能力成立。关系养成只有在生产路径回归和端到端验证都通过后，才能写成已实现结论。

## Implementation status

已完成：

- 试点生成改用“稳定核心 + 情境镜头”；完整人物卡只保留给正典校准和 Judge。
- 后续协议 0.4 已把稳定核心与动态情境镜头拆成独立 section，并增加由可信场景状态渲染的本轮回应合同；详见 [ADR-0009](0009-assemble-pilot-prompts-and-enforce-eval-gates.md)。
- 生产已确认 Memory 保留 memory id 与来源轮次，注入统一关系上下文合同。
- 试点 `RelationshipBranch` 通过适配器进入同一个关系上下文合同。
- 关系 Judge 必须逐字引用 R1/R2 回复片段并绑定实际提供的关系事件编号；代码会拒绝虚构片段或编号。
- 九场景协议升级为 0.3，旧产物不能复用为新协议结果。
- 当前评测协议已进一步升级为 0.4；0.3 结果只保留为历史对照，不能复用为 0.4 结果。

尚未完成：

- 真实 Turn API 仍使用旧版 16 人 `buildPersonaCard`；四位正典试点人物及其情境镜头尚未接入生产 `buildSystemBlocks`。
- `RelationshipBranch` 已以事件日志和可重建快照在生产 Store 中影子持久化，但尚未成为 Prompt 的正式关系来源。
- 生产仍按每个人物最近的已确认 Memory 加载，尚未完成相关性检索、冲突替代和过期策略。
- 试点评测仍需补真实 Turn API 的生产路径与端到端回归，才能验证线上产品表现。

## Consequences

- 人物完整度与单轮 Prompt 激活范围分离，减少“档案里有什么就每轮表演什么”。
- 生产和评测能用同一语言描述关系证据，降低 Prompt 层的双轨漂移。
- 当前实现不假装已经完成关系存储迁移；后续可在关系模型通过组件校准后逐步替换旧版 `intimacy` 和数组结构。
- 每次改变情境镜头映射或关系上下文合同都必须升级评测协议，并使用固定样本复测。
