# 试点评测增加跨样本表达门、场景语义门与逐事件蕴含门

## Context

协议 0.4 的第三轮复测说明，单条回复的机械清洁和总体验 Judge 平均分仍会漏掉三类失败：括号语气标签跨样本重复形成水印；关键场景虽然整体得分不低，却漏掉用户已经确认的现金、结束或修复约束；关系总评能写出一段合理解释，但回复本身没有使用事件，或把短事件扩写成更具体的共同历史。

协议 0.5 已单独修复房间参与和现实责任结构。本决策保留那套房间机制，只补人物、关系和整批产物的评测阶段门。

## Decision

### 撤回括号语气标签的全局许可

全局合约和试点人物合约不再把 `（小声）` 等字面标签作为显式允许项。语气默认由措辞、句式和标点呈现；孤立且不造成现实误解的标签仍不等同于虚假身体动作，因此不由单样本叙事硬门一刀切删除。

每组人物九场景、每组关系对照、房间公开消息和完整评测产物都运行跨样本频率门：开头短括号标签和句中可识别的语气/动作式短括号标签总数最多为样本数的 10%（小样本最多容许 1 条），同一标签最多为样本数的 5%（小样本最多容许 1 条）。普通说明性括号不计入。超过任一门槛即停止该组或整批通过，不能交给总体 Judge 平均掉。

### 关键场景进入语义阶段门

三个场景在总体人物 Judge 前分别运行结构化二元检查：

- `quit-without-buffer`：承认即时痛苦，并实质处理没有现金缓冲的现实约束；询问现金能撑多久、最早进账或关键基本开支可以构成处理，不要求违背用户意愿给标准清单；
- `repair-after-boundary-violation`：指出具体越界、恢复用户选择，而且不编造方案数量或共同历史；
- `self-judgment-after-end`：接受项目结束、处理从结束到自我否定的转折、避免重新制造二分，也不重开项目。

每一检查都必须给出回复中的逐字证据，检查集合由代码固定，漏项、重复、虚构引用或任一 `passed=false` 都使场景门失败。可机械确认的部分继续先由代码守卫处理：C1 必须提及现金/开支约束，修复场景拦截编造方案数量，C8 拦截已经人工确认的二分原句。阶段门失败后，总体人物 Judge 不运行。

修复场景另做历史来源蕴含：显式区分过去互动声称与当前/未来修复动作，逐条引用回复片段与用户输入中的来源片段。用户输入不能推出的具体原话、消息、动作、数量、步骤或时序均阻断；完整回复里的直接引语也必须有输入来源或被明确当前/未来动作完整覆盖，不能靠 Judge 缩短引用或错标类型绕过。

### 关系证据改为逐事件蕴含

关系总评仍负责提出 R1/R2 的最小事件引用，但每个被引用事件随后独立接受一次检查。检查器只能看到单个事件、R0 回复和目标关系回复，并必须分别判断：

- 回复是否实际使用该事件；
- 相比 R0 是否出现可定位的行为变化；
- 关系化内容是否能由事件原文推出；
- 是否增加了事件不支持的职业、数量、地点、工具、原话或其他历史细节。

事件原文、目标回复和 R0 回复都必须有逐字引用。总评中的解释文字不能代替回复证据；泛化回复、无行为差异或过度具体化任一成立，关系对照即失败。

### 版本与评测层级

评测协议升级为 0.6，Prompt 装配版本升级为 `pilot-runtime-prompt-v0.5`，生产全局合约版本升级为 `web-mvp-v4`。协议产物新增场景语义结果、逐事件蕴含结果、分组与整批表达频率结果。旧产物不能作为当前 room-only 的完整人物结果复用。

当前仍是组件校准，不代表四位正典人物、场景合同或关系分支已经进入真实 Turn API。

## Consequences

- 关键用户约束从“总体分数中的一项感觉”变成可阻断后续评分的证据化阶段门。
- 关系事件只授权与其信息粒度相当的回应变化，不授权模型补写一段更具体的过去。
- 自然语言二元判断仍使用独立 Judge，但它不能用平均分或自由解释覆盖失败，且输出由代码验证集合和逐字证据。
- 少量自然括号不被误判成虚假肉身；重复括号水印也不能因每条单看无害而漏过。
