# 已确认的决定权边界进入生产回应合同与输出硬门

## Context

生产 Turn 已能把确认式 Memory 注入关系上下文，但一次真实 `deepseek-v4-flash` 对话暴露了优先级冲突：用户边界是“可以给选项，但不要替用户决定”，偏好是“先给结论”，INTJ 最终直接回复“选压力大的”。Store、shadow 和 Prompt 证据均已到位；失败发生在模型把人格的结论先行和用户偏好排在关系边界之前。

仅把边界作为一条带来源的资料不足以保证行为。决定权又是可用窄规则机械识别的高影响边界，不应把最终安全性完全交给模型自觉或事后总体评分。

## Decision

- 关系上下文把已确认边界编译为本轮硬回应合同。边界优先于人格语气、用户偏好、主持器角度和直接给判断的冲动；偏好只能在边界内满足。
- 当边界明确保护用户的选择权时，Prompt 允许人物指出关键变量、比较标准、条件性后果或提出问题，禁止直接输出“选 X”“就选 X”“你应该选 X”等替用户拍板动作。
- 生产输出增加窄代码门，只识别已确认决定权边界下的直接选择动作。它不尝试通用理解所有自然语言边界，也不拦截条件分析、个人判断或把决定权留给用户的支持。
- 存在该硬边界时，人物输出先缓冲再校验，避免违规 token 已流给客户端。首版违规会携带具体重写合同再生成一次；最终仍违规则以已知失败终止，不返回越界文本。
- 一次后续真实创业选择对话虽然没有替用户决定，但以“你问反了”开场且只描述长期代价。决定权合同因此同时要求尊重用户提问，并必须给出本轮可用的比较方法；反模板门会让否定用户提问的压迫式开场重写。
- 当已确认偏好明确要求比较方法时，只有一个反问的回复不算完成，会触发一次反模板重写；已经给出方法后附带的问题仍然允许。
- 生产 Prompt 版本升级为 `web-mvp-v6`。

## Consequences

- 已确认的决定权边界不再与偏好平级，真实回复可以保留人物判断力而不替用户定夺。
- 受该边界保护的回复失去逐 token 首次流式展示，需要等窄硬门完成后一次性交付；这是防止越界文本泄露的有意权衡。
- 代码门保持保守和可解释，未覆盖的复杂语义仍需要评测与后续真实对话发现，不能宣称所有关系边界都已机械保证。
- “你问错了／问反了”只作为开场模式触发一次反模板重写；正文中的正常纠偏和不同意见不受影响。比较方法门只拦截可确定的“单句反问代替方法”，不会尝试用正则给任意自然语言方法评分；具体质量仍由真实对话评测。
